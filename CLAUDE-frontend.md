## 1. Project Overview

A production-grade, data-dense SPA for managing a multi-tenant OCI-compliant Docker
registry. The UI serves developers and DevOps engineers. The aesthetic is
**Modern Corporate / Technical** — precision over decoration, structural clarity over
visual noise, terminal-inspired data density.

### Design Personality (from DESIGN.md — treat as law)

> "Technical, reliable, and utilitarian — evoking the feeling of a well-organised
> terminal or a high-performance workstation."

Every component decision must pass this test:
**"Would a DevOps engineer trust this screen with production infrastructure?"**
If the answer involves decorative gradients, playful animations, or marketing copy,
reconsider.

## 3. Tech Stack — Final Decisions

These are fixed. Do not substitute without updating this file and the Decision Log.

| Concern | Package | Version | Reason |
|---|---|---|---|
| Framework | `react` | 19.x | Concurrent features, stable |
| Build tool | `vite` | 6.x | Fast HMR, static SPA output |
| Language | `typescript` | 5.x strict | Non-negotiable |
| Routing | `@tanstack/react-router` | 1.x | Type-safe params, nested layouts |
| Server state | `@tanstack/react-query` | 5.x | Background refetch, caching |
| UI primitives | `@radix-ui/*` via `shadcn/ui` | latest | Accessible, headless |
| Styling | `tailwindcss` | 4.x | Token-mapped to DESIGN.md |
| Tables | `@tanstack/react-table` | 8.x | Every screen is a data table |
| Charts | `recharts` | 2.x | Push frequency chart |
| Forms | `react-hook-form` + `zod` | latest | Type-safe validation |
| Client state | `zustand` | 5.x | Auth store, minimal global state |
| HTTP client | `axios` | 1.x | Interceptor support for JWT |
| Icons | `lucide-react` | latest | Matches design icon style |
| Fonts | `@fontsource/hanken-grotesk` + `@fontsource/jetbrains-mono` | latest | Exact DESIGN.md fonts, self-hosted |
| Date formatting | `date-fns` | 3.x | `formatDistanceToNow`, `format` |
| Toasts | `sonner` | latest | Minimal, non-intrusive |
| Testing | `vitest` + `@testing-library/react` | latest | Co-located with Vite |
| E2E | `playwright` | latest | Cross-browser |

### What is NOT used

- ❌ Next.js — no SSR needed, pure authenticated dashboard
- ❌ MUI / Ant Design / Chakra — too opinionated, fights the custom token system
- ❌ Redux / Jotai — TanStack Query handles server state; Zustand for the rest
- ❌ CSS Modules / styled-components — Tailwind only
- ❌ React Router — TanStack Router replaces it entirely
- ❌ Axios interceptors for retry — TanStack Query handles retry logic

### Token Usage Rules

- `bg-surface` — main application canvas (the outer `<body>` background)
- `bg-surface-container-lowest` / `border border-outline-variant` — cards
- `bg-surface-container-low` — sidebar background
- `bg-primary-container` — primary action buttons, active sidebar item background
- `text-on-surface` — primary body text
- `text-on-surface-variant` — secondary/muted text, column headers
- `text-on-tertiary-container` — `#009c54` green used for CLEAN / SUCCESS states
- `text-error` — CRITICAL severity, FAILED build status
- `text-severity-high` — HIGH severity (orange)
- `text-severity-medium` — MEDIUM severity (yellow)
- `font-mono` + `text-code-sm` or `text-code-md` — ALL technical data: digests, hashes,
  tags, version strings, docker commands, dockerfile snippets, build IDs
- `font-sans` + `text-label-caps` + `uppercase` + `tracking-[0.05em]` — table column
  headers, section labels

### Font Loading (`src/styles/globals.css`)

```css
@import '@fontsource/hanken-grotesk/400.css';

@import '@fontsource/jetbrains-mono/500.css';

@tailwind base;
@tailwind components;
@tailwind utilities;

@layer base {
  body {
    @apply bg-surface text-on-surface font-sans text-body-md antialiased;
  }
  code, pre, .mono {
    @apply font-mono;
  }
}
```

### Typography Rules

- **Never** use a system font for body text — Hanken Grotesk is always loaded first
- **Always** use JetBrains Mono for: SHA digests, docker commands, image tags, commit
  hashes, build IDs, version strings, dockerfile code, CLI output, any
  `<code>` element
- Table column headers: `text-label-caps uppercase tracking-[0.05em] font-bold
  text-on-surface-variant` — matches the screens exactly
- Page titles (`<h1>`): `text-display font-bold` on Build History screen;
  `text-headline-lg` on detail screens
- Stat card numbers: `text-[48px] font-black leading-none` for severity counts;
  `text-[28px] font-bold` for dashboard stats

## 6. Component Library

All primitive components come from shadcn/ui (copied into `src/components/ui/`, not
imported from a package). Customise them to match the token system — never override
with arbitrary values.

### 6.1 Button

Three variants, matching the screens exactly:

```tsx
// Primary — bg-primary-container, white text (Login button, Rescan, Manual Build)
<Button variant="primary">Login →</Button>

// Secondary / Ghost — border border-outline, white fill (Export Report, Filter Status)
<Button variant="ghost">Export Report</Button>

// Outline with icon
<Button variant="ghost">
  <RefreshCw className="w-4 h-4 mr-2" /> Rescan
</Button>
```

**Rules:**
- `rounded` (4px) for all buttons — not `rounded-lg`, not `rounded-full`
- `px-md py-sm` padding
- `hover:opacity-90 active:scale-95 transition-all` on primary buttons
- Never use `disabled` styling that only changes opacity — also add `cursor-not-allowed`
- Loading state: replace label with `<Loader2 className="animate-spin" />` + keep
  button width stable (use `min-w-*`)

### 6.2 StatusBadge

Used in the Repository Dashboard table and Build History table:

```tsx
type StatusVariant = 'healthy' | 'critical' | 'scanned' | 'in-progress' | 'success' | 'failed'

// Healthy: green dot + text
// 2 Critical: orange circle-exclamation + text
// Scanned: neutral pill
// In Progress: spinning icon + text
// Success: green checkmark + text
// Failed: red circle-x + text
```

**Rules:**
- `rounded-sm` (2px) — sharp rectangular badges, NOT pill-shaped (pills are for tags)
- Include an icon for accessibility (never colour-only status)
- Background: low-opacity of the status colour (`bg-error/10` for critical,
  `bg-tertiary-fixed/10` for success/healthy)
- Text: high-contrast version of the status colour

### 6.3 SeverityBadge

Used in the Security Scan Results and Image Details tables:

```tsx
type Severity = 'CRITICAL' | 'HIGH' | 'MEDIUM' | 'LOW' | 'NEGLIGIBLE'

// CRITICAL: bg-error/10 text-error border border-error/20
// HIGH:     bg-orange-50 text-orange-700 border border-orange-200
// MEDIUM:   bg-yellow-50 text-yellow-700 border border-yellow-200
// LOW:      bg-secondary-fixed text-on-secondary-fixed-variant border border-secondary/20
// NEGLIGIBLE: bg-surface-container text-on-surface-variant
```

Always include the severity icon (circle for critical, triangle for high/medium,
info circle for low).

### 6.4 TagChip

Image tags (`latest`, `v2.4.0-stable`, `beta-nightly`) styled as monospace chips:

```tsx
// bg-surface-container text-on-surface font-mono text-code-sm
// rounded (4px) px-sm py-[2px]
// For vulnerability-affected tags: bg-error/10 text-error
```

### 6.5 DigestChip

SHA256 digests truncated at 12 chars with tooltip showing full digest:

```tsx
// sha256:8f2a9...e31b
// bg-surface-container text-on-surface font-mono text-code-sm rounded px-sm py-xs
// Tooltip on hover showing full digest
// Copy to clipboard on click with sonner toast
```

### 6.6 CodeBlock

Dark-themed block for docker pull commands, dockerfile snippets, CLI output:

```tsx
// bg-[#1e293b] (slate-800) rounded-md p-lg
// Text: font-mono text-code-md text-[#e2e8f0]
// Syntax highlighting tokens:
//   commands (docker, FROM, RUN, COPY): text-[#7dd3fc]   (light blue)
//   strings/values:                      text-[#86efac]   (green — verified from screens)
//   comments/labels (DOCKERFILE header): text-on-surface-variant uppercase text-label-caps
// Copy button: absolute top-md right-md, lucide Copy icon, on click → useClipboard
```

**Do not** use a full syntax highlighting library (prism, shiki) for code blocks —
the designs show minimal two-tone highlighting only. Implement with `<span>` coloring.

### 6.7 DataTable

Generic wrapper around TanStack Table v8. Used on every authenticated screen.

**Required features:**
- Fixed header with `sticky top-0`
- Column headers: `text-label-caps uppercase text-on-surface-variant`
- Row hover: `hover:bg-surface-container-low transition-colors`
- Subtle zebra striping: even rows `bg-surface-container-lowest`, odd rows transparent
- Pagination: shown below table, format "Showing 1-20 of 1,242 builds"
- Empty state: `<EmptyState>` component, not an empty `<tbody>`
- Loading state: `<Skeleton>` rows matching column count
- No horizontal scroll on the outer container — tables scroll internally on mobile

```tsx
// Pagination controls match the design exactly:
// [ ← ] [ 1 ] [ 2 ] [ 3 ] [ → ]
// Active page: bg-primary-container text-on-primary rounded
// Inactive: text-on-surface-variant hover:bg-surface-container-low
```

## 10. State Management

### Principles

1. **Server state lives in TanStack Query.** Never duplicate API data in Zustand.
2. **Zustand only for:** auth state, user preferences (theme), UI state that must
   survive route changes (sidebar collapsed, notification panel open).
3. **Local component state** (`useState`) for: form state (handled by RHF anyway),
   modal open/close, tooltip visibility, table row selection.

### Auth Store (`src/store/authStore.ts`)

```typescript
interface AuthState {
  token: string | null           // JWT access token
  user: AuthUser | null          // decoded JWT payload
  tenantId: string | null        // current tenant ID
  setAuth: (token: string, user: AuthUser, tenantId: string) => void
  clearAuth: () => void
  isAuthenticated: () => boolean
}
```

**Rules:**
- JWT stored in memory only — **never** `localStorage` or `sessionStorage`
- Refresh token (if implemented) stored in HttpOnly cookie — never in JS
- On page reload: redirect to `/login`, do not attempt to restore from storage
- `clearAuth()` called on: logout, 401 response from API, token expiry

```typescript
})
```

## 11. API Integration Layer

### Axios Client (`src/lib/api/client.ts`)

```typescript
const apiClient = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 30_000,
  headers: { 'Content-Type': 'application/json' },
})

// Request interceptor: attach JWT
apiClient.interceptors.request.use((config) => {
  const token = useAuthStore.getState().token
  if (token) config.headers.Authorization = `Bearer ${token}`
  return config
})

// Response interceptor: handle 401
apiClient.interceptors.response.use(
  (res) => res,
  (error) => {
    if (error.response?.status === 401) {
      useAuthStore.getState().clearAuth()
      // TanStack Router navigate to /login
    }
    return Promise.reject(error)
  }
)
```

**Rules:**
- `VITE_API_BASE_URL` must be set — fail visibly in dev if missing
- Never log request/response bodies — they may contain tokens or image digests
- All API functions return typed responses (no `any`)
- Error responses mapped to typed `ApiError` — never expose raw axios errors to UI

## 12. Authentication Flow

### JWT Flow (credential login)

```
1. User submits login form
2. POST /auth/token { credential, password }
3. Response: { access_token, tenant_id, user }
4. Store token in Zustand memory store
5. Navigate to /repos
6. Every API request: Authorization: Bearer <token>
7. On 401: clearAuth() → redirect to /login
8. Token expiry (300s): re-prompt login (no silent refresh for now)
```

### SSO Flow

```
1. User clicks "Login with SSO"
2. POST /auth/sso/initiate { tenant_domain }
3. Response: { redirect_url }
4. window.location.href = redirect_url
5. SSO provider redirects back to /auth/sso/callback?code=xxx
6. Exchange code: POST /auth/sso/callback { code }
7. Store token, navigate to /repos
```

### Route Guard

```typescript
// src/routes/_authenticated/route.tsx
export const Route = createFileRoute('/_authenticated')({
  beforeLoad: ({ context }) => {
    if (!context.auth.isAuthenticated()) {
      throw redirect({ to: '/login' })
    }
  },
})
```

## 13. Security Rules

### XSS Prevention

- Never use `dangerouslySetInnerHTML` — zero exceptions
- All user-controlled content (repo names, descriptions, tag names) rendered as text
  nodes only, never injected as HTML
- CVE descriptions and scan findings rendered as plain text — never trust backend
  HTML content
- Docker pull commands in `<CodeBlock>` constructed client-side from typed fields,
  never rendered from a raw backend string

### Content Security Policy

Set in `index.html` (development) and via Nginx headers (production):

```
Content-Security-Policy:
  default-src 'self';
  script-src 'self';
  style-src 'self' 'unsafe-inline';
  font-src 'self';
  img-src 'self' data:;
  connect-src 'self' <VITE_API_BASE_URL>;
  frame-ancestors 'none';
```

**Note:** `unsafe-inline` for styles is required by Tailwind v4's runtime CSS
injection in development. Production builds emit static CSS — remove `unsafe-inline`
in production CSP.

### Token Handling

- JWT in Zustand memory only — wiped on page reload (by design)
- No `Authorization` header logged in browser DevTools Network tab (use `sensitive`
  header flag where supported)
- Refresh tokens (future): HttpOnly, Secure, SameSite=Strict cookie only

### External Links

All external links (CVE advisories, documentation):
- `target="_blank" rel="noopener noreferrer"`
- Open in new tab, never navigate away from the app

### Input Handling

- All form inputs validated with Zod before submission
- Repo name, tag name, org name validated against the same regex patterns as the
  backend (see backend `claude.md` §7 Input Validation)
- Do not display raw API error messages to users — map to friendly messages

## 14. Accessibility Requirements

- All interactive elements reachable via keyboard Tab order
- All icon-only buttons have `aria-label`
- `<StatusBadge>` and `<SeverityBadge>` include icon + text — never colour-only status
- `<DataTable>` uses semantic `<table>`, `<thead>`, `<th scope="col">`, `<tbody>`, `<tr>`, `<td>`
- `<Dialog>` / modal components trap focus (Radix handles this automatically)
- `prefers-reduced-motion`: no transforms or animations when set. Use:
  ```css
  @media (prefers-reduced-motion: reduce) {
    *, *::before, *::after { animation-duration: 0.01ms !important; transition-duration: 0.01ms !important; }
  }
  ```
- Colour contrast: all text meets WCAG 2.1 AA (4.5:1 for body, 3:1 for large text)
- `<DigestChip>` tooltip: accessible via keyboard focus, not just hover
- Loading states use `aria-busy="true"` on the container
- Error states use `role="alert"` for screen readers

```nginx
  add_header Referrer-Policy "strict-origin-when-cross-origin" always;
  add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self'; font-src 'self'; img-src 'self' data:; connect-src 'self'; frame-ancestors 'none';" always;
  add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

  # No cache for index.html (ensures new deployments are picked up)
  location = /index.html {
    add_header Cache-Control "no-cache, no-store, must-revalidate";
  }

  # Long cache for hashed assets
  location ~* \.(js|css|woff2|png|ico)$ {
    expires 1y;
    add_header Cache-Control "public, immutable";
  }
}
```

## 17. Coding Conventions

### TypeScript

- `strict: true` in `tsconfig.json` — no exceptions
- No `any` — use `unknown` and narrow, or define a proper type
- All React components: typed props interface, no prop spreading onto DOM elements
- All API response types defined in `src/types/api.ts` — mirror the backend Go structs

### React

- Functional components only — no class components
- Custom hooks for all business logic — components are display-only where possible
- `useCallback` and `useMemo` only when measurably needed — not by default
- Fragment shorthand `<>` preferred over `React.Fragment`

### File Naming

- Components: `PascalCase.tsx`
- Hooks: `camelCase.ts` with `use` prefix
- Utilities: `camelCase.ts`
- Types: `camelCase.ts`
- Route files: TanStack Router conventions (see §2)

### Import Order (enforced by ESLint)

```typescript
// 1. React
import { useState, useEffect } from 'react'
// 2. Third-party
import { useQuery } from '@tanstack/react-query'
// 3. Internal absolute (@/)
import { DataTable } from '@/components/data/DataTable'
import { queryKeys } from '@/lib/queryKeys'
// 4. Relative
import { columns } from './columns'
```

### Tailwind Class Order

Use `prettier-plugin-tailwindcss` to enforce consistent class ordering.

### Component Documentation

Every component in `src/components/registry/` must have a JSDoc comment block:
```typescript
/**
 * SeverityBadge — displays a CVE severity level with icon and colour coding.
 * Maps directly to the Security Scan Results table in the Stitch designs.
 * @see design/stitch/security_scan_results/screen.png
 */
```

| 1 | React + Vite over Next.js | Pure authenticated SPA — no SSR benefit, simpler deployment, static Nginx serving | Initial |

| 7 | No syntax highlighting library | Designs show two-tone minimal colouring only — Prism/Shiki overkill, adds bundle weight | Initial |
| 8 | @fontsource over Google Fonts CDN | Self-hosted = no external request = CSP `font-src 'self'` achievable | Initial |
| 9 | Zustand over Redux | Auth store is the only global state needed — Zustand is 2KB vs Redux's complexity | Initial |
| 10 | Recharts over Chart.js / Victory | Best React integration, responsive by default, matches bar chart style in designs | Initial |
| 11 | No Angular / Vue | React chosen: TanStack ecosystem (Table, Query, Router) has deepest React support | Initial |
| 12 | Material Symbols → Lucide React | Designs use Material Symbols; Lucide is a React-native equivalent with matching style and tree-shaking | Initial |
